# KOKUSEI Development Guardrails

このファイルはリポジトリ全体に適用する。AIエージェントと人間の開発者は、変更前に本書と`README.md`を確認する。

# Product Principles

- KOKUSEIのミッションは「数字の先にある、日本の構造へ。」
- 公的機関の一次情報をもとに、日本の構造を理解するためのサービスとする。
- 利用者へ結論を与えず、自分で考えるための判断材料を提供する。
- 推測・意見・解釈を事実として表示しない。
- 機能追加は「日本の構造を理解する助けになるか」を判断基準とする。
- この目的に関係しない機能は追加しない。

# Development Rules

- 指示されていない機能は追加しない。
- 既存のDocker Compose環境を壊さない。
- Frontend（表示）、Backend（API・ユースケース・取得処理）、Database（永続化）の責務を維持する。
- API互換性を破る変更は、実装前に影響範囲を説明する。
- DBスキーマ、認証、データモデルなどの大きな設計変更は、実装前に方針を提示して確認を求める。
- 意図を左右する不明点は推測で実装せず、確認する。

# Data Rules

- 公的な一次情報のみを公式データとして扱う。
- 出典URL、公開日、取得日時を保持する。
- 統計値を`float64`で扱わない。PostgreSQLの`numeric`精度を維持し、APIでは10進文字列として返す。
- 外部データ取得に失敗しても、既存データを削除・上書きしない。
- 開発用データ、概算値、確定値、公式データを明確に区別する。
- データ加工や単位変換は、内容を記録・表示できる設計にする。
- 外部取得は`IndicatorDataProvider`、保存はrepositoryに分離し、取得レスポンスを検証してから保存する。

# Security

- `.env`、APIキー、認証情報、`DATABASE_URL`などの秘密情報をGitへコミットしない。
- 公開する環境変数ファイルは`.env.example`のみとする。
- 秘密情報をログ、README、テスト、エラーメッセージへ出力しない。
- 秘密情報を検査するときは値そのものを表示しない。

# Code Quality

コードは「動けばよい」ではなく、将来AIと人間の両方が読みやすいことを重視する。

- シンプルな設計を優先し、過度な抽象化を避ける。
- 小さく変更し、小さくテストする。
- コメントに頼らず、名前で意図が伝わるコードを書く。
- 同じロジックを重複実装しない。
- 可読性を最優先する。

# Required Verification

変更内容に応じて、該当する検証を実行する。

## Backend

- `gofmt`
- `go vet ./...`
- `go test ./...`

## Frontend

- `npm run lint`
- `npm test`
- `npm run build`

## Integration

- Docker Composeの起動確認
- `GET /health`のヘルスチェック
- 主要画面の表示確認
- DB変更時のmigration適用・rollback確認

リポジトリの`Makefile`にあるコマンドを優先して利用する。実行できなかった検証を成功と報告せず、理由を明記する。

# Completion Report

作業完了時は必ず以下を報告する。

- 実装内容
- 変更した主要ファイル
- 設計判断
- 実行したテストと結果
- 実行できなかった検証
- 残課題・既知のリスク

## 承認前の必須報告

実装完了後、ユーザーに承認を求める前に、既存の完了報告を含めて以下を必ず報告する。

1. 変更の目的
   - この変更で実現したことを3行以内で説明する。
2. 変更内容
   - 変更した主要ファイル、追加・変更・削除した機能、既存仕様への影響を示す。
3. 動作確認
   - 実行したテストと結果、実行していないテストとその理由を示す。
4. リスク
   - 壊れる可能性がある箇所、後方互換性への影響、セキュリティ・性能・データ損失に関する懸念を示す。
5. 未確認事項
   - 確認できていない点、推測や仮定に基づいて実装した点、自信が低い判断を示す。
6. 承認時にユーザーが確認すべきこと
   - 確認すべき画面、操作、仕様を具体的に示す。

- 「問題ありません」「完了しました」だけで承認を求めない。
- テスト未実施、仕様不明、既存機能への影響が不明な状態を、承認可能と表現しない。
- 事実として確認できた内容と、推測・仮定に基づく内容を明確に分ける。
- 問題や未確認事項が残っている場合は隠さず報告する。
- 承認を得ること自体を目的にせず、ユーザーが根拠を確認して判断できる情報を提示する。

## 承認の種類ごとの報告

### 1. 作業途中の承認

コマンド実行、権限昇格、外部アクセス、破壊的操作などの承認を求める前に、以下を簡潔に説明する。

- 実行する内容と、その必要性
- 変更される可能性があるもの
- 失敗した場合の影響
- 読み取り専用か、変更を伴う操作か
- より安全な代替手段の有無

### 2. 実装完了時の承認

「承認前の必須報告」の全項目を提示する。

### 3. 本番反映・リリース時の承認

通常の報告と「承認前の必須報告」に加え、以下を必ず提示する。

- 本番環境への影響
- データ変更の有無
- ロールバック方法
- ロールバックできない変更の有無
- リリース後の確認方法
- 監視すべきメトリクスやログ
- 障害発生時の初動
